  • Provide excellent developer experience/ergonomics.
  • Be lazy loaded is core of the architecture. When a component first hits the DOM we need to either wire it immediatly or download the cooresponding class code and then wire it up.
  • Be type safe - export types so that if a dev wants to use a component with any framework + typescript it should also allow type safety.
  • Be as tiny as possible with as few polyfills as possible.
  • Components should be able to work with future versions of the stencil compiler with as few breaking changes as possible. We wanted the ability to run components written with older version of Stencil to be able to be included in projects with new versions of Stencil.

    • Component API

  • Component - Class decorator that accepts general metadata or information about the component

    • Component API

  • Prop - Property Decorator - designates component attributes/properties that can be passed to this component... with Types. If the prop passed to the component changes then the component rerenders and updates the visual representation of the component
  • State - Property Decorator - designates which class property is part of the components state. The component reacts to changes in state and forces a rerender.

    • Component API

  • Watch - Method Decorator - designates a function to be called when the given state/prop name changes.
  • componentWillLoad is one of the many lifecycle methods. This one is called right before the component has loaded

    • Component API

  • render Method - returns a JSX representation of what should appear on the page. This method was reduced for brevity JSX can seem foreign but it is super expressive and supported by TypeScript out of the box.

    • How the sausage gets made

    Metadata Collection


  • There is alot of information defined by developers while writing the components that can provide us with valuable information while transforming and generating the resulting code.
  • This is how you begin to use the TS Compiler API. Create a program and pass it your source files
  • Then we use its typechecker and begin walking the AST of all files provided. Whats interesting about how these tools is that we can do some nice type analysis in the TS API.

    • Metadata Collection


  • Examples would be tagName, all style data, what are possible properties, well mostly all decorators.
  • Also test for valid data, we depend on types but sometimes you need more validation
  • We infer things based on your prop types. This allows us to be more effecient in our runtime code.
  • We walk the AST and collect the information and normalize the data based on some usage rules.
  • All the data we collected during the metadata collection lets add that back to the component in a straight forward way

    • Power of Transforms

  • After we have collected all of the metadata it is time to start adapting the code
  • I will be showing the before/after later during the demo
  • We have a number of AST transforms that let us reduce the complexity of code before TypeScript does its own transpliation, as well as some that we use after.
  • Again we are going to use the TS API. Each transform listed here runs over the AST of every file included in the TS build.

    • Power of Transforms

  • Removes all decorators that are specific to stencil, We don't need these at runtime we have already collected all of the metadata
  • We will then add all the component metadata collected back in but as accessor properties on the class

    • Power of Transforms

  • If all components in your project do not use specific features then really non of your runtime needs code around supporting those features. This allows us to reduce the amount of common code that needs to be shipped to a browser.

    • Power of Transforms

  • Purge anything left that we do not need really the removal at these steps are safeguards
  • Collect some more information after transpile about relationships between components that can allow the building to intelligently group components together when downloaded.

    • Seamless* Upgrades

  • Component code compiles down to what we call a standardized target component. It is normalized across projects and potentially across versions of Stencil.
  • We write codemods for these standardized target components so that if a component written in version 0.0.7 needs to be changed to work with 0.0.8 then we can upgrade the component at compile time.
  • These situations would occur if you found a standalone component that you like and wanted to included in your own internal component collection you would not need multiple stencil runtimes delivered to a browser.
  • Controlling the output of our compiled code and versions to our compiler means we know exactly how the api changed and then the codemods that need to occur to ensure it works with newer runtime code.
  • The asterisk is because have had to create 2 codemods so far and I cannot see that far into the future but we will try.
